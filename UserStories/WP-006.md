# WP-006: Create path parameters

- **ID**: WP-006
- **Description**: As a technical user, I want to define and manage path parameters (like {id} in /users/{id}), so I can properly document API endpoints with dynamic segments.
- **Acceptance criteria**:

  - When user enters a path with placeholders (e.g., /users/{id}), the system automatically recognizes each placeholder as a path parameter
  - Placeholders must be enclosed in curly braces: {paramName}
  - The system validates that multiple parameters are separated by path segments: /user/{id}/{name} is valid but /user/{id}{name} is invalid
  - Invalid placeholder format (missing braces, adjacent parameters) triggers a validation error with clear guidance
  - A "Path parameters" section appears between the path input field and the operations list when a path contains parameters
  - Path parameters section only appears when path parameters exist (no section shown for paths without parameters)
  - Path parameters section displays a clean, organized list of all detected parameters from the path
  - Parameters are listed in the same order they appear in the path
  - Each parameter row displays: parameter name, a type selector dropdown, and a description field
  - Type selector offers OpenAPI base parameter types: string, number, integer, boolean, array (at minimum)
  - Each parameter has a text input field for adding/editing a description
  - When user modifies the path and adds a new parameter placeholder, the new parameter automatically appears in the path parameters section
  - When user modifies the path and removes a parameter placeholder, the corresponding parameter is automatically removed from the path parameters section
  - Removed parameters do not trigger warnings or confirmations (auto-removal is transparent)
  - Real-time validation ensures all parameter names in the section match the placeholders in the path
  - Real-time validation prevents duplicate parameter names
  - If user manually deletes a parameter from the section but the placeholder remains in the path, a validation warning appears
  - Parameter name validation rejects invalid characters and shows helpful error message
  - Type selection is mandatory for each parameter and defaults to "string"
  - Description field is optional and can be left empty
  - Users can edit parameter descriptions inline by clicking on the description text
  - Changes to type and description are automatically saved to local storage
  - Path parameters are included in the exported YAML following OpenAPI specification format
  - Each parameter is marked as "required=true" in the OpenAPI output (path parameters are always required)
  - When user selects a different path from the sidebar, the path parameters section updates to show parameters for the newly selected path
  - The path parameters section maintains consistent styling and layout with the rest of the form
  - Visual hierarchy clearly indicates the path parameters section is between the path input and operations list
  - All parameter data is persisted to local storage alongside the path data
  - Users can manage parameters for multiple different paths without data loss or conflicts
